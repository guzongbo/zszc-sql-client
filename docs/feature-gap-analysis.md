# 功能差距分析

本文档不再按“通用数据库客户端大而全对标”来罗列功能，而是基于当前仓库真实能力，分析 `zszc-sql-client` 作为一款 **MySQL 客户端桌面应用** 还缺哪些关键能力，以及哪些方向暂时不适合前置投入。

## 结论先行

- 当前产品最有辨识度的能力不是“多数据库支持”，而是 MySQL 场景下的数据对比、结构对比和本地桌面工作流。
- 因此下一阶段最值得投入的，不是立刻扩成通用数据库平台，而是先补齐专业客户端的基础短板：
  - 安全连接
  - 凭据保护
  - 查询结果导出
  - 草稿与会话恢复
  - 结构对比任务化
- 多数据库支持、ER 图、可视化查询构建器这类方向并非完全没价值，但在当前阶段都不应排在前面。

## 现有能力盘点

以下内容是当前仓库中已经有代码支撑的能力。

### 1. 连接与数据源管理

- 支持 MySQL 连接配置的新增、编辑、删除、测试和分组管理。
- 支持从 Navicat `.ncx` 导入连接配置。
- 支持浏览数据库列表和表列表。

### 2. 表结构与表数据操作

- 支持创建数据库。
- 支持创建表、读取表结构、编辑表结构、查看表 DDL。
- 支持分页浏览表数据。
- 支持基于 `WHERE` / `ORDER BY` 的表数据筛选。
- 支持对主键表进行新增、更新、删除，并能预览将要执行的 SQL。

需要纠正的点：

- 当前写入流程虽然会在后端使用事务提交，但还没有真正提供“手动事务会话”能力。
- 前端存在 `transaction_mode` 字段，但没有形成前后端闭环，不能算成熟功能。

### 3. SQL 控制台

- 内置 Monaco 编辑器。
- 支持 SQL 自动补全元数据加载。
- 支持执行查询和变更语句。
- 查询结果支持分页加载。

### 4. 数据对比与结构对比

- 数据对比支持：
  - 表发现
  - 后台任务执行
  - 进度查询
  - 取消任务
  - 差异详情分页
  - 导出同步 SQL
  - 本地历史记录
- 结构对比支持：
  - 对比执行
  - 详情查看
  - 导出结构 SQL
  - 本地历史记录

需要纠正的点：

- 结构对比目前还不是任务化执行，没有进度查询和取消能力。

## 已有但未闭环的缺口

这些问题不一定都是“新功能”，但都直接影响产品成熟度。

- 连接信息当前保存在本地 SQLite 中，密码为明文存储，存在明显安全风险。
- 连接模型只有 `host/port/username/password`，没有 SSH 隧道、SSL/TLS 等生产环境常见能力。
- UI 中已有“导出表”“复制表”“删除表”等入口或占位，但并未真正完成闭环。
- 没有 SQL 草稿、控制台标签恢复、查询历史、未保存工作区恢复。
- 数据对比已任务化，结构对比仍是同步等待，核心体验不一致。

## 功能差距与优先级

### 高优先级

#### 1. 安全连接与凭据保护

建议内容：

- 支持 SSH 隧道连接
- 支持 SSL/TLS 连接参数
- 连接密码改为系统钥匙串或本地加密存储
- 提供“记住密码”和“仅本次使用”之类的控制项

理由：

- 这是桌面数据库客户端进入真实工作场景的门槛能力。
- 当前明文密码存储风险过高，优先级高于大多数效率功能。
- 没有 SSH/SSL，很多测试环境和生产环境根本无法接入。

#### 2. 查询结果与表数据导出

建议内容：

- 表数据导出 CSV
- 查询结果导出 CSV / JSON
- 明确导出范围：
  - 当前页
  - 当前筛选结果
  - 用户选中行

理由：

- 当前已经支持导出对比 SQL，但不支持把查询结果本身导出出来，实际工作流不完整。
- 前端已有“导出表”入口占位，补齐这条链路能快速提升可用性。

#### 3. 草稿、标签恢复与查询历史

建议内容：

- 保存 SQL 控制台草稿
- 恢复上次打开的标签和当前数据库上下文
- 保存表数据筛选条件
- 保存最近执行 SQL 历史

理由：

- 这是桌面应用区别于临时网页工具的重要体验。
- 项目约定里已经明确本地 SQLite 优先承载草稿、历史、缓存，这部分应尽快落地。

#### 4. 结构对比任务化

建议内容：

- 为结构对比补齐 `start / progress / result / cancel`
- 统一与数据对比的任务反馈模型
- 前端复用同一套进度展示与轮询逻辑

理由：

- 数据对比和结构对比属于同一核心卖点，但当前交互体验不一致。
- 结构对比一旦遇到大库或大表数量，缺少进度和取消会明显影响可用性。

### 中优先级

#### 1. 数据导入向导

建议内容：

- 支持 CSV / JSON 导入表数据
- 提供字段映射、空值处理、预览和失败反馈

理由：

- 这与导出能力形成完整闭环。
- 对测试造数、一次性迁移、小批量修复都很实用。

#### 2. SQL 执行安全与分析能力

建议内容：

- 危险 SQL 二次确认
- `EXPLAIN` / 执行计划查看
- 查询结果行数限制预设

理由：

- 当前控制台已具备基本执行能力，但还缺少更专业的保护和分析工具。
- 这比“可视化查询构建器”更贴近当前产品目标用户。

#### 3. 对象级常用操作闭环

建议内容：

- 完成“导出表”
- 实现表复制
- 明确是否支持删表，如果支持则补齐确认、校验和错误反馈

理由：

- 这些入口当前已经出现在界面中，用户会自然期待它们可用。
- 先把已有入口补完整，比继续添加新菜单更合理。

### 低优先级

#### 1. SQL 片段 / 模板

适合重度 SQL 用户，但不是当前最阻塞的问题。

#### 2. 用户与权限图形化管理

更偏 DBA 使用场景，等连接安全、导入导出、工作区持久化稳定后再考虑。

#### 3. ER 图查看

有价值，但当前还不是核心短板。相较之下，安全连接和工作流持久化更影响日常使用。

## 当前不建议前置投入的方向

### 1. 多数据库支持（PostgreSQL / SQLite）

这一项在通用竞品里很常见，但不适合当前阶段排高优先级，原因如下：

- 项目定位仍是 MySQL 客户端桌面应用。
- 当前连接模型、DDL 处理、对比逻辑、自动补全、表设计和本地存储都偏向 MySQL。
- 一旦引入多数据库支持，后端抽象、前端表单、结构对比语义和测试矩阵都会显著膨胀。

更合理的顺序是：

1. 先把 MySQL 主线能力做深
2. 把安全、导出、持久化、任务体验补齐
3. 再根据真实用户需求判断是否扩数据库种类

### 2. 可视化查询构建器

这类能力实现复杂，收益又高度依赖交互设计成熟度。当前先把 SQL 控制台、安全保护、历史和草稿做好，更符合项目阶段。

### 3. 备份与恢复图形化向导

有一定价值，但在当前阶段优先级低于连接安全、数据导出和结构对比体验统一。

## 推荐路线

1. 先补安全连接与凭据保护。
2. 再补结果导出和本地工作区持久化。
3. 同步完成结构对比任务化。
4. 然后再做数据导入、执行计划和对象级操作闭环。
5. 最后再评估多数据库支持、ER 图或可视化查询构建器。

这样更符合当前仓库的真实能力边界，也更符合 `zszc-sql-client` 现阶段“先做强 MySQL 桌面客户端主线”的产品路线。
