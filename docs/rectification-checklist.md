# 项目整改清单

本文档用于跟踪 `zszc-sql-client` 当前阶段的整改顺序、优先级和推进状态。原则是先处理高风险和高回报问题，再处理结构优化与性能深挖，避免同时做大面积重构导致回归风险失控。

## 当前推进策略

1. 先收安全暴露面。
2. 再收口前端与桌面端的职责边界。
3. 然后处理重复实现与热点性能问题。
4. 最后补齐质量门禁与持续验证。

## P0

- [x] 建立整改清单并锁定优先级顺序
- [x] 收敛连接凭据暴露链路
  目标：
  - 启动引导数据不再直接把已保存密码下发到前端
  - 编辑已有连接时允许密码留空并复用本地已保存密码
  - Redis 连接列表也不再默认回传密码
- [x] 收敛数据对比缓存的数据落盘风险
  目标：
  - 梳理 diff cache 的生命周期
  - 对任务完成后的缓存清理建立明确策略
  - 为后续“禁用明细落盘/仅保留摘要”预留开关
- [x] 修复当前静态检查失败项
  目标：
  - 前端 lint 归零
  - 收敛 Rust `clippy` 的结构性告警
  - 为后续把 clippy 纳入门禁做准备

## P1

- [ ] 拆分前端总控壳层
  目标：
  - `App.tsx` 收敛为装配层
  - 数据源、对比、历史、弹窗、输出面板分域管理
- [x] 统一长任务模型
  目标：
  - 结构对比补齐 `start / progress / result / cancel`
  - 统一前端轮询和结果读取逻辑
- [ ] 抽离重复实现
  目标：
  - MySQL 访问能力避免同步/异步双栈长期并行扩散
  - 数据源分组维护 UI 在 MySQL/Redis 工作区之间复用
  - mock 语义与桌面端真实语义保持一致
- [ ] 修正明显状态正确性问题
  目标：
  - 标签关闭、切换、批量更新避免使用陈旧闭包状态
  - 高频状态更新不再牵动整棵树重渲染

## P2

- [x] 优化 Redis key 扫描链路
  目标：
  - 减少 `SCAN + TYPE + TTL` 的 N+1 请求模式
  - 为大 key 空间场景补分页与批量元数据策略
- [x] 优化历史记录读取模型
  目标：
  - summary/detail 分离
  - 避免列表接口反序列化整块大 JSON
- [ ] 优化前端包体与重资源加载
  目标：
  - 继续压缩 Monaco 相关 chunk
  - 验证是否需要更细粒度延迟加载
- [x] 补齐质量门禁
  目标：
  - CI 改为常规触发
  - 纳入 lint、test、clippy

## 当前执行中

- 批次 2：可靠性与状态正确性
  已完成：
  - 连接密码不再随 bootstrap / 列表接口默认下发
  - 已有连接编辑时空密码保留原值
  - 现有前端 lint 报错已清零
  - 标签关闭时不再依赖旧 `tabs` 闭包状态
  - 数据对比已完成任务增加保留期清理，避免结果长期堆积在内存
  - 数据对比结果切换和页面卸载时，会主动清理上一份 compare cache
  - `App.tsx` 中数据源树、对比表过滤、历史筛选、插件工作区选项等高频派生值已做稳定化，减少无意义重算
  - Redis key 扫描从逐 key `TYPE + TTL` 改为批量管道查询，减少大 key 空间下的往返开销
  - diff cache 新增摘要模式与保留期环境开关，默认行为不变，按需可关闭明细落盘
  - Rust `clippy --all-targets -D warnings` 已清零，并同步纳入 CI 常规触发
  - 工作区切换器已从 `App.tsx` 抽离为独立组件，继续压缩壳层职责
  - 历史记录列表已改为摘要读取，详情按 id 懒加载，列表页不再反序列化大 JSON
  - SQL 编辑器入口在控制台和数据表页进一步下沉为懒加载包装，减少非编辑场景的 Monaco 装载压力
  - 结构对比已补齐 `start / progress / result / cancel` 任务接口，前端不再单独维护同步直跑链路
  - 数据对比与结构对比共用同一套任务轮询 helper，减少重复状态逻辑和维护分叉
  下一步候选：
  - `App.tsx` 壳层拆分
  - Monaco chunk 进一步拆分
  - 数据源分组维护 UI 继续抽共享
