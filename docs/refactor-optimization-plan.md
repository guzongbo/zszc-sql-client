# 项目重构与性能优化方案

本文档基于当前仓库实际代码状态，给出 `zszc-sql-client` 的重构与性能优化建议。目标不是一次性“推倒重来”，而是在不打断现有功能迭代的前提下，把前端和 Tauri 命令层逐步收敛到可维护、可扩展、可观测的结构。

## 结论先行

- 前端重构方向是合理的。`frontend/src/App.tsx` 当前约 8352 行，页面渲染、状态管理、命令调用、表格编辑、数据对比、结构对比和历史记录都堆在一个入口文件里，已经超过可维护边界。
- 但当前最优先的问题不是“立刻全面引入 Zustand 和虚拟列表”，而是先拆职责、再收敛状态、最后按热点做性能优化。
- 后端长任务统一方向也合理，但不建议为结构对比再复制一套专用任务管理器，更适合抽象成通用任务运行时。

## 现状判断依据

### 前端

- `frontend/src` 目前仍是扁平结构，核心逻辑主要集中在 `App.tsx`、`SqlEditor.tsx`、`api.ts`、`types.ts`。
- 表数据页与 SQL 控制台已经有分页能力：
  - 表数据默认分页加载，后端默认 `limit=100`，最大 500。
  - SQL 控制台默认分页加载，后端默认 `limit=200`，最大 500。
  - 数据对比详情按页加载，前端一次请求 50 条，并有本地缓存上限。
- 因此，“大列表卡顿”的风险真实存在，但还没有到所有列表都必须优先虚拟化的程度。

### 后端

- 数据对比已经采用后台任务模型：`compare_start / compare_progress / compare_result / compare_cancel`。
- 结构对比仍是直接执行的 `structure_compare_run`，没有任务 ID、没有进度查询、没有取消机制。
- 当前 `CompareTaskManager` 是面向数据对比结果定制的，不适合再复制一份同构实现。

## 当前方案中不够合理的部分

### 1. 直接把“引入 Zustand”当作第一步，风险偏高

当前 `App.tsx` 的问题首先是职责混杂，不是单纯“缺一个全局状态库”。如果在没有拆分领域边界前就把大量状态整体搬进 Store，会把原本难以理解的本地状态，变成更难追踪的全局状态。

更稳妥的顺序应该是：

1. 先把领域逻辑拆出 `App.tsx`
2. 再识别哪些状态确实跨模块共享
3. 最后再决定是否引入 Zustand

### 2. 把“全面虚拟列表”列为优先项，依据不足

当前仓库里最重的结果集场景已经有分页和分段加载，直接上全面虚拟化会增加实现复杂度、滚动同步成本和调试成本，但收益未必最大。

更合理的做法是：

- 先解决根组件大面积重渲染
- 先解决重复计算和状态耦合
- 只对经验证的热点区域做虚拟化

### 3. 为结构对比单独再建一个 `StructureCompareTaskManager`，会制造重复维护

数据对比和结构对比在任务生命周期上是同一类问题：注册、执行、汇报进度、取消、完成、取结果。这里更适合抽象“通用任务壳层”，而不是再复制一套结构相似的管理器。

## 建议的重构顺序

### 阶段一：先做职责拆分，不改用户行为

目标：把 `App.tsx` 从“总控脚本”拆成“页面装配层”。

建议优先抽离以下模块：

- `frontend/src/features/datasource`
  - 连接配置、分组、树节点展开、连接状态
- `frontend/src/features/table-data`
  - 表数据加载、分页、行编辑、SQL 预览、提交
- `frontend/src/features/sql-console`
  - 控制台 SQL、结果集、分页、补全缓存
- `frontend/src/features/data-compare`
  - 数据对比表单、任务轮询、详情分页、导出 SQL
- `frontend/src/features/structure-compare`
  - 结构对比表单、结果、详情、导出 SQL
- `frontend/src/features/history`
  - 对比历史筛选、选择、详情展示
- `frontend/src/shared`
  - 通用 UI、小工具函数、格式化函数、空状态、状态面板

这一阶段优先抽：

- 纯类型和常量
- 纯工具函数
- API 组装逻辑
- 与单一领域绑定的自定义 Hook
- 已经在 `App.tsx` 底部定义的大块局部组件

不建议这一阶段做的事：

- 改接口协议
- 大改视觉结构
- 同时引入多个状态管理方案

### 阶段二：收敛状态模型，减少根组件重渲染

目标：让状态按领域归属，避免 `App.tsx` 对所有局部状态都持有写权限。

建议策略：

- 每个业务域先收口到自己的 `useReducer` 或领域 Hook。
- 真正跨域共享的状态再考虑上提，例如：
  - 引导数据与连接配置缓存
  - 数据源树展开状态
  - SQL 自动补全缓存
  - 对比历史列表
  - 长任务轮询状态
- Tab 内部编辑态尽量继续保持局部，不要急着全局化。

如果第二阶段完成后仍存在以下问题，再引入 Zustand：

- 同一份状态被多个无亲缘组件频繁读写
- 提升状态层级导致 props 传递过深
- 轮询状态和缓存状态在多个流程中重复实现

如果引入 Zustand，建议只承载“共享缓存”和“任务状态”，不要把所有表单输入和编辑中的临时值一股脑搬进去。

### 阶段三：按热点做性能优化，而不是先上重型手段

当前优先级更高的性能项如下：

#### 1. 降低派生数据重复计算

重点检查：

- 数据源分组、排序、筛选是否在每次渲染时全量重算
- 对比详情切换时是否重复构造大对象
- 历史记录和下拉选项是否重复排序

可用手段：

- 抽出纯计算函数
- 在明确热点上使用 `useDeferredValue`
- 对切换大型结果视图的交互使用 `startTransition`

#### 2. 缩小状态更新影响范围

重点检查：

- 轮询数据对比进度时，是否导致与当前页面无关的区域一起重渲染
- 打开/关闭弹窗是否影响整棵组件树
- SQL 编辑器与结果视图是否被无关状态反复刷新

#### 3. 只对确认的热点引入虚拟化

建议把虚拟化放到以下场景的后置选项中：

- 数据源树节点规模明显增大时
- 对比详情缓存上限放宽后，单次展示行数明显提升时
- 历史记录列表达到长列表级别时

不建议把已经有分页的表数据页和 SQL 结果页作为第一批虚拟化目标。

### 阶段四：统一长任务模型

目标：数据对比和结构对比对前端暴露一致的任务接口。

建议改造方向：

- 抽象通用任务运行时，例如：
  - `register`
  - `report_progress`
  - `finish_success`
  - `finish_failure`
  - `request_cancel`
  - `take_result`
- 在此基础上为结构对比补齐：
  - `structure_compare_start`
  - `structure_compare_progress`
  - `structure_compare_result`
  - `structure_compare_cancel`
- 统一前端轮询协议和进度展示组件，避免数据对比、结构对比各写一套状态机。

如果后续还有导出、批量同步等耗时任务，这套模型也可以复用。

### 阶段五：顺手清理当前的“半成品状态”

这部分不是纯性能问题，但会持续拉低可维护性，建议在重构过程中一并处理：

- 梳理 `transaction_mode` 这类前后端未闭环字段，避免文档和代码语义不一致。
- 把“未开发占位按钮”与真实能力区分清楚，避免 UI 暗示错误能力。
- 给对比、导出、表编辑等关键流程补最小回归测试或最小自测清单。

## 推荐实施顺序

1. 拆出 `features` 目录和领域 Hook，先把 `App.tsx` 压到可读范围。
2. 把共享缓存和轮询状态从页面局部逻辑中分离出来。
3. 统一结构对比与数据对比的长任务模型。
4. 对已确认热点做定向性能优化。
5. 最后再决定是否引入 Zustand，以及是否需要局部虚拟化。

## 预期收益

- `App.tsx` 从超大入口文件收敛为装配层，后续页面迭代风险下降。
- 数据对比、结构对比、表数据、SQL 控制台形成清晰边界，便于并行开发。
- 轮询和进度模型统一后，前后端接口更稳定，用户体验也更一致。
- 性能优化聚焦真实热点，避免为了“看起来现代化”引入不必要复杂度。
