# 项目重构与性能优化方案

本文档旨在分析当前 `zszc-sql-client` 项目在代码结构和性能方面存在的问题，并提出一系列可行的重构与优化建议。

## 核心问题诊断

项目当前的核心问题主要集中在**前端架构**和**后端长任务处理**两个方面。虽然核心的 Rust 对比引擎 (`compare_core`) 已经采用了高效策略，但应用上层的封装和交互逻辑成为了性能和可维护性的主要瓶颈。

---

## 优化方案详情

### 1. 前端：巨型单体组件 `App.tsx` 重构

- **问题现状：**
  前端主组件 `frontend/src/App.tsx` 的代码量超过 8000 行，几乎管理了整个应用的所有状态（`useState`）和 UI 逻辑。这是一个巨大的性能隐患和维护性灾难。任何微小的状态变更都可能触发整个 `App` 组件及其庞大的子组件树进行不必要的 Diff 和渲染。

- **优化目标：**
  将 `App.tsx` 拆分为多个独立、可复用、易于管理的子组件，并引入现代化的状态管理机制，实现精确、高效的渲染控制。

- **实施步骤：**
  1.  **组件拆分：** 将 `App.tsx` 按照功能域拆分为更小、更专注的子组件。建议拆分出以下模块：
      - `DataSourcePanel`: 负责左侧数据源树的展示和交互。
      - `Workspace`: 负责管理所有标签页（Tabs）的布局和状态。
      - `DataCompareWorkflow`: 封装整个数据对比流程（从选择数据源到展示结果的三个步骤）。
      - `StructureCompareWorkflow`: 封装整个结构对比流程。
      - `DataTab`, `DesignTab`, `ConsoleTab`: 为每种 Tab 类型创建独立的、功能内聚的组件。
  2.  **状态管理中心化：** 引入专业的状态管理库，推荐 **Zustand**（因其轻量级和对 Hooks 的良好支持，非常适合此项目）。
      - 将全局状态（如：所有连接配置、数据源分组）和复杂的领域状态（如：数据对比、结构对比的工作流状态）从 `App.tsx` 的 `useState` 中剥离出来，放入共享的 Store 中。
      - 组件可以按需订阅它们关心的状态，而不是在根组件中接收所有状态的更新，从而实现更精确的渲染。
  3.  **使用 React Context：** 对于作用域相对独立的模块（例如某个特定的弹窗状态），可以使用 `React.Context` 结合 `useReducer` 来管理其内部状态，避免全局状态污染。

### 2. 前端：大数据列表渲染虚拟化

- **问题现状：**
  在数据展示场景，如“数据表内容”、“SQL 查询结果”、“数据对比详情”等，代码直接将一整页的数据行渲染到 DOM 中。当一页显示较多行数（例如 1000 行）时，会创建大量的 DOM 节点，导致页面滚动卡顿、内存占用飙升。

- **优化目标：**
  实现无论数据总量多大，都能流畅滚动的列表视图。

- **实施步骤：**
  1.  **引入虚拟列表（List Virtualization）：** 对所有可能展示大量数据的表格/列表，使用虚拟化技术进行重构。推荐的开源库：
      - **TanStack Virtual** (`@tanstack/react-virtual`)
      - `react-window`
  2.  **实现原理：** 虚拟列表只渲染当前视口（viewport）内可见的列表项。当用户滚动时，动态更新和复用这些列表项。这可以确保 DOM 中始终只维持几十个节点，从根本上解决大数据渲染的性能问题。

### 3. 后端：长任务处理模型统一

- **问题现状：**
  在 `commands.rs` 中，对长任务的处理方式不统一。
  - **数据对比 (`compare_start`)：** 实现优秀。通过 `tauri::async_runtime::spawn` 将对比任务派发到后台线程执行，并有任务管理器（`compare_tasks`）来跟踪进度和处理取消请求。
  - **结构对比 (`structure_compare_run`)：** 实现较为简单。它是一个普通的 `async` 命令，整个对比过程会在 Tauri 的命令处理线程上执行，缺乏独立的任务管理、进度回报和取消功能。

- **优化目标：**
  统一所有耗时操作的后端处理模型，为用户提供一致的（可取消、带进度条的）交互体验。

- **实施步骤：**
  1.  **统一任务模型：** 仿照 `compare_start` 的模式重构 `structure_compare_run`。
  2.  **创建 `StructureCompareTaskManager`：** 在 Rust 后端建立一个用于管理结构对比任务的管理器，类似于现有的 `compare_tasks`。
  3.  **提供进度和取消 API：** 创建新的 `structure_compare_start` 命令，用于启动任务并立即返回任务 ID。同时，提供 `structure_compare_progress` 和 `structure_compare_cancel` 命令，让前端可以轮询进度和请求取消任务。这不仅优化了后端的并发处理能力，也极大地提升了用户体验。
